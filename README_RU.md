# Gyre

[English](README.md) | [简体中文](README_ZH.md) | **Русский**

> **Высокопроизводительный агентный фреймворк на Rust** — объединяет высший уровень познания и управления процессами из Zoo-Code и точность низкоуровневой манипуляции кодом из oh-my-pi, реализуя современный AI-агент для программирования полностью в экосистеме Rust.

## 📋 Ключевые возможности

Gyre — это **продакшен-фреймворк для AI-агента-программиста**, обеспечивающий замкнутый цикл *восприятие → рассуждение → действие*:

### 🤖 Движок агента
- **Пяти-состоянийный конечный автомат** (NoTask → Running → Streaming → WaitingForInput → Idle): перенят из зрелой модели состояний Zoo-Code для точного управления жизненным циклом агента.
- **Архитектура, управляемая сообщениями**: модели сообщений `say` (информационные) / `ask` (интерактивно-блокирующие) со шлюзом одобрения и квитанциями для пользователя.
- **Несколько режимов**: `code` (написание кода) / `architect` (архитектура) / `ask` (вопрос-ответ) / `debug` (отладка). Режим определяет системный промпт и подмножество инструментов.
- **Механизм steering**: внедрение сообщений в процессе выполнения для динамического вмешательства в направление рассуждений агента.
- **Мягкие требования к инструментам**: сначала предупреждение, затем принуждение — для защиты префиксного кэша провайдера.

### 🛠️ Инструментарий
- **Базовые инструменты**: `read_file` / `write_file` / `str_replace` / `apply_diff` / `list_files` / `run_command` / `grep` / `glob`
- **AST-манипуляция кодом** (опционально): `replace_block` (замена синтаксического блока tree-sitter) / `ast_search` / `ast_rewrite` (структурный поиск и перезапись через ast-grep)
- **Интеграция с LSP** (опционально): диагностика / переход к определению / поиск ссылок / переименование символа
- **Hashline DSL** (опционально): линейно-привязанное пакетное редактирование, применение нескольких файлов и диапазонов за один проход
- **PTY псевдотерминал** (опционально): `run_pty_command`, поддержка интерактивных терминальных приложений (top / vim / REPL)
- **Обработка изображений** (опционально): `read_image` / `image_gen` (DALL·E или совместимый API)
- **Интеграция с GitHub** (опционально): запросы и операции с PR / Issues / Actions, а также GraphQL-запросы
- **Протокол MCP**: совместим с Model Context Protocol; инструменты любого MCP-сервера можно подключить

### 🧠 Управление контекстом
- **AppendOnlyLog**: сообщения доступны только для добавления; единственный легальный путь изменения — это `replaceTail` во время уплотнения
- **Отпечатки StablePrefix**: системный промпт + спецификация инструментов делаются моментальным снимком один раз и фиксируются байтовым отпечатком SHA-256 для максимального попадания в префиксный кэш на стороне провайдера
- **Трёх-уровневое уплотнение**: Shake (удаление избыточности по дрожанию) → Summarize (LLM-резюме для передачи) → Prune (резерв по окну), в сочетании с Tool-Protection для защиты результатов инструментов
- **Подсчёт токенов**: на базе `tiktoken-rs`, точное отслеживание использования по каждому ходу

### 🔌 Мультипровайдер LLM
- **ProviderRegistry**: единая точка входа маршрутизирует по `model.api`, со встроенной аутентификацией `ProviderCallContext` и ограничением конкурентности
- **OpenAI Chat Completions**: охватывает OpenAI / совместимые шлюзы / локальный vLLM / Ollama
- **DeepSeek**: DeepSeek Chat / Reasoner
- **GLM (Zhipu / Z.ai)**: полная поддержка официального API — переключатель `thinking`, эхо размышлений `reasoning_content`, многоходовое продолжение `preserveReasoning`
- **Открытый адаптер**: реализуйте трейт `LlmProvider` + регистрация одной строкой для добавления нового провайдера
- **Режим размышления** (Thinking/Reasoning): поддержка бюджета токенов размышления для `deepseek-reasoner` / моделей GLM с размышлением (glm-4.7+/glm-5+)

### 🌐 Два фронтенда
- **CLI / REPL**: построчное редактирование `rustyline` + автодополнение Tab + потоковый рендеринг Markdown + подсветка синтаксиса
- **Веб-интерфейс** (React + TypeScript + Vite): двусторонняя WebSocket-коммуникация в реальном времени, оптимизированный по трафику проводной протокол (дельта-приоритет)
- **Управление сессиями**: персистентная история в JSONL, поддержка возобновления / ветвления / экспорта
- **Многоязычность (i18n)**: CLI и веб-интерфейс по умолчанию следуют системному языку; встроены китайский / English / Русский / 日本語; переключение в настройках или конфиге; добавление языка требует лишь одного файла ресурсов

### 🔌 Протокол ACP (интеграция с редакторами)
- **Agent Client Protocol v1**: следует стандарту [agentclientprotocol.com](https://agentclientprotocol.com), предоставляя интерфейс JSON-RPC 2.0 + SSE/stdio для совместимых клиентов, таких как Zed Editor
- **Два режима транспорта**: `stdio` (редактор вызывает как подпроцесс, нулевое использование портов) / `http` (HTTP+SSE, мультиплексируется на том же порту, что и веб-фронтенд)
- **Тройная эквивалентность**: ACP использует то же управление сессиями и механизм одобрения, что и CLI и веб-фронтенд — одну сессию могут одновременно отслеживать несколько клиентов
- **Стандартный поток событий**: `session/update` отправляет стандартные события ACP, такие как `agent_message_chunk` / `agent_thought_chunk` / `tool_call` / `usage_update`
- **Объявление возможностей**: рукопожатие `initialize` достоверно объявляет `promptCapabilities` (поддержка изображений) и `mcpCapabilities` (поддержка stdio), проходя строгую валидацию схемы клиентом

### 🔄 Оркестрация мульти-агентов
- **Делегирование под-агентов** (TaskTool): родительский агент может делегировать подзадачи под-агентам, с ограничителями конкурентности и независимыми бюджетами токенов
- **Swarm мульти-агент**: рабочие процессы DAG, определяемые в YAML, параллельное выполнение внутри волны, обратные вызовы прогресса конвейера
- **Мониторинг Supervisor**: отслеживание в реальном времени состояний всех под-агентов (выполняется/успех/сбой) через CLI-дашборд и веб-панель

### 🤝 Ретрансляция совместной работы
- **Сквозное шифрование**: запечатанные байты AES-256-GCM; ретрансляционный сервер «слеп» (маршрутизирует только по room_id)
- **WebSocket-мост**: несколько пользователей могут присоединиться к одной комнате для совместного использования сессии агента в реальном времени

### 📐 Архитектурный дизайн
- **Ports & Adapters (Гексагональная архитектура)**: все основные зависимости внедряются как `Arc<dyn Trait>`; цикл полностью прозрачен для транспорта / провайдера / инструментов
- **Неинвазивное расширение**: саморегистрация плагинов на этапе компиляции через `inventory`; добавление провайдера/инструмента не требует изменений центрального реестра
- **Однонаправленные зависимости**: `core` не имеет бизнес-зависимостей; `agent` зависит только от трейтов, а не от конкретных реализаций; граф зависимостей однонаправленный и ацикличный
- **Событийно-ориентированность**: цикл порождает `Stream<Item = AgentEvent>`; оба фронтенда подписываются единообразно

## 🚀 Быстрый старт

### Предварительные требования

- Инструментарий Rust 1.85+ (см. [`rust-toolchain.toml`](rust-toolchain.toml))
- API-ключ LLM (OpenAI / Anthropic / DeepSeek / GLM / локальный vLLM)

### Установка и запуск

```bash
# Клонировать репозиторий
git clone https://github.com/Gyre/Gyre.git
cd Gyre

# Скопировать конфиг и вписать API-ключ
cp config.example.toml .agent/config.toml

# Сборка (режим Release)
cargo build --release

# Запустить разовую задачу
./target/release/agent "Выведи список файлов в текущем каталоге в виде дерева"

# Или запустить интерактивный REPL
./target/release/agent

# Запустить веб-сервис
./target/release/agent --serve
```

### Конфигурация

Конфигурация использует TOML и загружается по слоям (последующий переопределяет предыдущий):

1. **Пользовательский уровень**: `~/.config/agent/config.toml` (Windows `%APPDATA%\agent`)
2. **Уровень проекта**: `<cwd>/.agent/config.toml`

Поддерживается несколько профилей моделей; переключение во время выполнения через `--model <alias>` или команду `/model <alias>`:

```toml
[default_model]
id               = "gpt-4o-mini"
api              = "openai-completions"
base_url         = "https://api.openai.com/v1"
api_key          = "${OPENAI_API_KEY}"
temperature      = 0.2
max_output_tokens = 4096

[[models]]
id        = "deepseek-chat"
alias     = "ds"
api       = "openai-completions"
base_url  = "https://api.deepseek.com"
api_key   = "${DEEPSEEK_API_KEY}"
```

Полный набор опций см. в [`config.example.toml`](config.example.toml).

### 🌐 Многоязычность (i18n)

CLI и веб-интерфейс **по умолчанию следуют системному языку** (CLI читает `LANG` / `LC_ALL` / `LC_MESSAGES`; веб — `navigator.languages`), со встроенными 4 языками:

| Код | Язык    | Ресурс CLI | Ресурс веб |
| ---- | ------- | ---------- | ---------- |
| `en` | English | [`crates/i18n/locales/en.json`](crates/i18n/locales/en.json) | [`web/c5-ui/src/lib/locales.ts`](web/c5-ui/src/lib/locales.ts) |
| `zh` | 中文    | `crates/i18n/locales/zh.json` | он же |
| `ru` | Русский | `crates/i18n/locales/ru.json` | он же |
| `ja` | 日本語  | `crates/i18n/locales/ja.json` | он же |

**Переопределение в конфиге** (опционально; оставьте пустым для автоопределения системного языка):

```toml
# Верхний уровень config.toml
language = "zh"   # en / zh / ru / ja
```

- **CLI**: [`main.rs`](crates/cli/src/main.rs) инициализируется системным языком при запуске (`agent_i18n::init(None)`); после загрузки конфига явное `language` переопределяет его.
- **Веб**: переключение на [панели настроек](web/c5-ui/src/components/agent/SettingsPanel.tsx) в разделе «Оформление → Язык»; предпочтение сохраняется в `localStorage`, а режим «Auto» следует системе и мгновенно обновляется при смене системного языка.

#### Добавление языка (симметрично в двух местах)

1. **CLI**: скопируйте `en.json` в `<code>.json` в [`crates/i18n/locales/`](crates/i18n/locales/) и переведите; затем зарегистрируйте по одной строке в `LOCALE_FILES`, `SUPPORTED` и `match_supported` внутри [`lib.rs`](crates/i18n/src/lib.rs).
2. **Веб**: добавьте `Dict` в [`locales.ts`](web/c5-ui/src/lib/locales.ts), включите его в `locales`/`SUPPORTED_LOCALES` и заполните собственное отображаемое имя через `lang.<code>` в каждом языке.

Цепочка отката единообразна: текущий язык → английский → сам ключ (отсутствующие ключи видны при разработке без ошибок).

```bash

agent

# Выполнить задачу
> Проанализируй src/main.rs и оптимизируй производительность

# Уплотнить контекст вручную
/compact

# Показать использование
/status

# Оркестрация мульти-агентов Swarm
/swarm workflow.yaml
```

### Веб-сервис

```bash
# Прослушивать адрес [server].bind из конфига (по умолчанию 127.0.0.1:8080)
agent --serve
# → http://127.0.0.1:8080

# Указать адрес привязки (переопределяет конфиг)

```

### 🔌 Протокол ACP и интеграция с редакторами

Gyre реализует сервер [Agent Client Protocol (ACP) v1](https://agentclientprotocol.com), открывая возможности агента совместимым клиентам, таким как Zed Editor. ACP использует то же управление сессиями и механизм одобрения, что и CLI и веб-фронтенд — одну сессию могут одновременно отслеживать несколько клиентов.

#### Режимы транспорта

| Режим | Способ запуска | Сценарий применения |
| ---- | -------------- | ------------------- |
| **stdio** | `agent --acp-stdio` | Редактор запускает как подпроцесс (рекомендуется; нулевое использование портов; процесс завершается автоматически при выходе редактора) |
| **http** | `agent --serve --acp` | HTTP+SSE, мультиплексируется на том же порту, что и веб-фронтенд; подходит для кастомных фронтендов или удалённого доступа |

#### Режим stdio

Редактор запускает agent как подпроцесс и обменивается JSON-RPC-сообщениями, разделёнными переносом строки, через stdin/stdout. Дополнительный порт не нужен.

```bash
# Запустить напрямую (ручное тестирование)
agent --acp-stdio
# Вводите JSON-RPC в stdin; ответы и уведомления session/update возвращаются в stdout
```

#### Режим HTTP+SSE

Использует тот же HTTP-сервис, что и веб-фронтенд, монтируя два эндпоинта:

| Эндпоинт | Метод | Описание |
| ---- | ---- | ---- |
| `/acp/rpc` | POST | Точка входа JSON-RPC 2.0-запросов (`initialize` / `session/new` / `session/prompt` …) |
| `/acp/sse/{session_id}` | GET | SSE-поток событий; подписка на уведомления `session/update` для заданной сессии |

```bash
# Включить ACP HTTP-эндпоинты (также включить постоянно через [acp] enabled = true в config.toml)
agent --serve --acp

# При запуске выводит:
#   • ACP  JSON-RPC  POST /acp/rpc
#            SSE events  GET  /acp/sse/{session_id}
```

> **Аутентификация**: режим HTTP использует `[server] auth_token` (тот же токен, что и у веб-фронтенда). Поддерживается заголовок `Authorization: Bearer <token>` или параметр запроса `?token=<token>`. Если `auth_token` не задан, валидация не выполняется.

#### Конфигурация

Управляется через секцию `[acp]` в `config.toml` (соответствует `crates/acp`):

```toml
[acp]
enabled   = false             # выключен по умолчанию; true автоматически монтирует маршруты /acp/* при --serve
transport = "http"            # "http" (HTTP+SSE) / "stdio" / "both"
```

CLI-флаги переопределяют конфиг во время выполнения: `--acp` (включить HTTP+SSE), `--acp-stdio` (только stdio; высший приоритет; не запускает HTTP).

#### Конфигурация Zed Editor

Зарегистрируйте Gyre как ACP-агента в `settings.json` Zed (**режим stdio**, рекомендуется):

```json
{
  "agent": {
    "enabled": true,
    "profiles": {
      "Gyre": {
        "name": "Agent Project",
        "transport": {
          "kind": "stdio",
          "command": "/path/to/agent",
          "args": ["--acp-stdio"],
          "env": {}
        },
        "default_model": {
          "provider": "custom",
          "name": "Gyre"
        }
      }
    },
    "default_profile": "Gyre"
  }
}
```

> **Путь**: замените `/path/to/agent` на фактический путь вашего артефакта сборки (например, `./target/release/agent`, или просто `agent` после установки на `$PATH`).
>
> **Конфигурация модели**: сервер ACP повторно использует профили `[default_model]` и `[[models]]` из `config.toml`, загружая ту же маршрутизацию провайдеров. Вам не нужно повторно вводить API-ключи в редакторе — выберите алиас модели через параметр `model` (или `_meta.model`) на `session/new`.

При использовании **режима HTTP** (agent работает постоянно с `--serve --acp`) конфиг Zed принимает вид:

```json
{
  "agent": {
    "enabled": true,
    "profiles": {
      "Gyre": {
        "name": "Agent Project",
        "transport": {
          "kind": "http",
          "url": "http://127.0.0.1:8080/acp/rpc"
        }
      }
    },
    "default_profile": "Gyre"
  }
}
```

#### Поддерживаемые RPC-методы

| Метод | Тип | Описание |
| ---- | ---- | ---- |
| `initialize` | Запрос | Рукопожатие + согласование возможностей (возвращает `protocolVersion` / `agentCapabilities` / `agentInfo`) |
| `session/new` | Запрос | Создать новую сессию; возвращает `{ sessionId }`; указать начальную модель и режим через параметры `model` / `mode` (или `_meta`) |
| `session/prompt` | Запрос | Отправить сообщение пользователя; блокирует до завершения хода и возвращает `{ stopReason }`; в процессе постоянно отправляет уведомления `session/update` |
| `session/cancel` | Уведомление | Отменить выполняемый ход |
| `session/load` | Запрос | Возобновить историческую сессию (передать существующий `sessionId`); возвращает новый `sessionId` |
| `session/close` | Запрос | Закрыть и освободить ресурсы сессии |
| `session/set_mode` | Запрос | Переключить режим агента (`code` / `architect` / `ask` / `debug`) |

#### Типы событий session/update

Стандартные события ACP, отправляемые во время `session/prompt` через SSE (режим HTTP) или stdout (режим stdio):

| `sessionUpdate` | Описание |
| --------------- | ---- |
| `agent_message_chunk` | Инкрементальный текст ответа агента |
| `agent_thought_chunk` | Инкрементальные размышления / reasoning агента |
| `tool_call` | Создание вызова инструмента (включает `toolCallId` / `title` / `kind` / `status` / `rawOutput`) |
| `tool_call_update` | Обновление статуса / вывода вызова инструмента |
| `usage_update` | Обновление использования окна контекста (`used` / `size`) |

#### Объявление возможностей (ответ initialize)

```json
{
  "protocolVersion": 1,
  "agentCapabilities": {
    "loadSession": true,
    "promptCapabilities": { "image": true, "embeddedContext": false },
    "mcpCapabilities": { "http": false, "stdio": true }
  },
  "agentInfo": { "name": "Gyre", "version": "0.1.0" },
  "authMethods": []
}
```

> `promptCapabilities` и `mcpCapabilities` — обязательные поля спецификации ACP v1; строгие клиенты (такие как Zed) валидируют схему и отклоняют рукопожатие при их отсутствии.

#### Быстрая проверка

```bash
# 1. Ручное тестирование режима stdio (ввод JSON-RPC построчно)
agent --acp-stdio
# Ввод:
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":1,"clientInfo":{"name":"test","version":"1.0"}}}
# → возвращает ответ agentCapabilities

# 2. Ручное тестирование режима HTTP
agent --serve --acp &
# Создать сессию
curl -X POST http://127.0.0.1:8080/acp/rpc \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"session/new","params":{"model":"ds","mode":"code"}}'
# → {"jsonrpc":"2.0","id":1,"result":{"sessionId":"..."}}
```

Дизайн и реализация этого проекта глубоко вдохновлены и опираются на следующие проекты с открытым исходным кодом:

#### [Zoo-Code](https://github.com/nicepkg/zoo-code) (концептуальная ссылка)

Zoo-Code предоставил ключевой проектный чертёж для **высокоуровневого агентного познания и управления процессами**. Этот проект заимствовал следующие ключевые концепции:

- **Пяти-состоянийный конечный автомат** (NoTask → Running → Streaming → WaitingForInput → Idle): точное моделирование жизненного цикла агента
- **Модель сообщений say/ask**: чёткое разделение информационного вывода и интерактивно-блокирующих запросов
- **Маршрутизация одобрения (AskDispatcher)**: механизм координации между потоковым выводом и пользовательским вводом
- **Система режимов** (code / architect / ask / debug): конфигурация системного промпта и подмножества инструментов для разных сценариев
- **Абстракция провайдера и преобразование форматов**: преобразование форматов и стратегии кэширования для проводных протоколов разных вендоров

#### [oh-my-pi](https://github.com/nicepkg/oh-my-pi) (концептуальная ссылка + миграция дизайна)

oh-my-pi привнёс зрелые практики в **точность низкоуровневой манипуляции кодом и инженеризацию цикла выполнения**. Этот проект **полностью перенёс** его гибридную архитектуру TypeScript/Rust **в чистый Rust**:

- **Цикл выполнения** (agent-loop.ts): полный замкнутый цикл — потоковые рассуждения → разбор вызовов инструментов → пакетное выполнение → обратная загрузка результатов
- **AppendOnlyLog + StablePrefix**: память контекста и максимизация попаданий в префиксный кэш провайдера
- **Подсистема уплотнения** (summarization / pruning / shake / tool-protection): управление окном контекста
- **LLM-диспетчеризация** (ProviderRegistry / streamSimple): маршрутизация к нескольким провайдерам через единую точку входа
- **AST-манипуляция** (pi-ast): структурные операции над кодом на базе tree-sitter + ast-grep
- **Нативный инструментарий** (pi-natives): низкоуровневые возможности, такие как grep / glob / highlight / tokens
- **Многослойная конфигурация** (settings-schema.ts): глубокое слияние конфигураций пользовательского и проектного уровней
- **Движок правил одобрения** (approval.ts): правила allow/deny/ask по каждому инструменту + сопоставление glob на уровне команд
- **Оптимизированный по трафику проводной протокол**: WebSocket-коммуникация с приоритетом дельт
